# Resumo Executivo

Este relatorio analisa como diferentes familias de backends analiticos e incrementais se comportam quando usadas como camada de serving para paineis gerenciais near-real-time orientados por eventos. O problema avaliado nao se reduz a latencia SQL isolada; ele envolve a distancia entre o evento canonico de negocio e a disponibilidade de um estado derivado semantica e contabilmente confiavel no frontend. Para enfrentar esse problema, o documento consolida um benchmark reprodutivel sobre um repositorio monolitico que integra producer, OpenTelemetry Collector, Kafka, storage writer, APIs especializadas, gateway realtime e frontends por backend. O corpus tecnologico comparado abrange ClickHouse, Apache Druid, Apache Pinot e Materialize, cobrindo tanto motores hot-analytic quanto sistemas incremental-streaming. A metodologia experimental opera por rodadas individualizadas, com coleta de metricas de bootstrap, latencia de API, latencia SQL, convergencia websocket, corretude contabil e consumo de recursos, com os budgets de fase explicitamente registrados em `metadata.json` de cada rodada. A bateria conclusiva real, executada no commit `2211abae82145de28056003773f83d8785fe454e`, mostrou que ClickHouse apresentou o menor p95 de API para `summary` (58.26 ms), Pinot apresentou os menores p95 de SQL (`entries_count` em 3.04 ms e `summary_by_role` em 3.32 ms), Druid atingiu `frontend_time_to_first_meaningful_state_ms` de 45 ms com `snapshot_rate_per_second` de 2.307, enquanto Materialize preservou corretude contabil total, mas com maior custo de consulta e maior volatilidade operacional (`health_transition_total` igual a 14). Em todas as stacks observadas, `balance_sheet_difference` permaneceu em 0.0. O resultado principal deste relatorio e demonstrar, com dados reais e rastreaveis, que nao ha dominancia universal entre paradigmas de serving: o backend superior depende da dimensao observada e do peso atribuido a latencia de leitura, convergencia visual e estabilidade operacional.

**Palavras-chave:** benchmark de backends, dashboards em tempo real, event sourcing, OLAP, incremental view maintenance, Materialize, ClickHouse, Druid, Pinot.

# Executive Summary

This report analyzes how different families of analytical and incremental backends behave when used as the serving layer for event-driven near-real-time management dashboards. The evaluated problem is not limited to isolated SQL latency; it concerns the distance between the canonical business event and the availability of a semantically and financially reliable derived state in the frontend. To address this problem, the document consolidates a reproducible benchmark built on top of a monorepo integrating producer, OpenTelemetry Collector, Kafka, storage writer, specialized APIs, realtime gateway and backend-specific frontends. The technological corpus includes ClickHouse, Apache Druid, Apache Pinot and Materialize, therefore covering both hot-analytic engines and incremental-streaming systems. The experimental methodology is organized as isolated rounds, collecting metrics for bootstrap, API latency, SQL latency, websocket convergence, accounting correctness and resource consumption, with phase budgets explicitly persisted in each round metadata. The final real benchmark battery, executed on commit `2211abae82145de28056003773f83d8785fe454e`, showed that ClickHouse achieved the lowest API p95 for `summary` (58.26 ms), Pinot achieved the lowest SQL p95 values (`entries_count` at 3.04 ms and `summary_by_role` at 3.32 ms), Druid reached a `frontend_time_to_first_meaningful_state_ms` of 45 ms together with a `snapshot_rate_per_second` of 2.307, while Materialize preserved full accounting correctness but at the highest query cost and with the highest operational volatility (`health_transition_total` equal to 14). In all observed stacks, `balance_sheet_difference` remained 0.0. The main contribution of this report is to show, with real and auditable data, that there is no universal winner among serving paradigms: the preferable backend depends on how one prioritizes read latency, visual convergence and operational stability.

**Keywords:** backend benchmarking, real-time dashboards, event sourcing, OLAP, incremental view maintenance, Materialize, ClickHouse, Druid, Pinot.
